# HTTPS 란 무엇인가?

웹 사이트를 이용하다보면 아래와 같이 크롬이 경고하는 것을 본 적이 있을 것 입니다. 이것은 HTTPS 프로토콜을 사용하지 않기 때문에 발생하는 페이지입니다.  ![image](https://user-images.githubusercontent.com/52944973/176989263-bbf737c5-0bcb-4216-8ed5-8fbe47b41529.png)

그렇다면 HTTPS 가 무엇이길래 크롬에서 HTTPS 를 사용하지 않는 웹사이트에는 경고 창 까지 띄워주는 것 일까?

먼저 HTTP 에 대해서 알아보겠습니다. **HTTP** 란 인터넷에서 **텍스트로 이루어진 HTML** 과 같은 문서를 서버와 클라이언트가 주고 받을 때 사용하는 통신 프로토콜 입니다.  HTTP 서버는 기본 포트인 80번 포트를 사용합니다.

**HTTPS** 는 Hyper Text Transfer Protocol over **Secure Sockets Layer** 의 약자로 HTTP 통신에서 **소켓통신 레이어에** 안전성을 더해준다는 의미입니다.





# SSL 이란?
**Secure Sockets Layer** 를 줄여서 **SSL** 이라고 하는데요, SSL 이란 간단하게 말해서 HTTP 에서 주고 받는 텍스트 형태로 이루어진 데이터를 암호화하는 보안 기능을 갖고 있는 **보안 인증서** 입니다.

SSL 은 OSI 7 계층에서 응용 계층과 전송 계층 사이에 존재합니다. 데이터를 전송할 때는 SSL 계층에서 데이터를 암호화해서 하위 계층으로 보내고 데이터를 받을 때는 SSL 계층에서 복호화하여 상위 계층으로 보냅니다.

# 대칭키/비대칭키

흔히 암호화 할때 대칭키 방식과 비대칭키 방식을 설명합니다. 
- **대칭키 방식** : 보내고자 하는 데이터를 하나의 키로 암호화하고 복호화하는 방식입니다. A 라는 데이터를 암호화 할 때 B 라는 키를 사용한다면 복호화 할 때도 B 라는 키를 사용하는 것 입니다. 이 경우 해커가 대칭키를 알게 된다면 보안이 뚫려버린다는 단점이 존재합니다.
- **비대칭키 방식** : A 와 B 가 통신을 한다고 가정했을 때, A가 B 에게 공개키를 알려줍니다. B 는 A 의 **공개키**로 암호화해서 A 에게 데이터를 보냅니다. A 는 **비밀키**로 해당 데이터를 복호화 합니다. A 의 **공개키**로 암호화된 데이터는 A 의 **비밀키**로만 복호화 할 수 있고 A 의 **비밀키**는 A 만 알고 있기 때문에 상당히 **안전**하다는 장점이 있습니다.

비대칭키 방식이 무조건 좋아보이지만 비대칭키 방식은 컴퓨팅 파워가 많이 듭니다.

따라서 HTTPS 는 **대칭키 + 비대칭** 키 방식을 사용합니다. **비대칭키를 활용해서 대칭키를 안전하게 주고받고 그 뒤로 대칭키를 통해서 통신합니다.**

# TSL 란?
SSL 과 TLS 는 같은 뜻으로 흔히들 말합니다. 사실 TLS 는 SSL 의 새로운 버전인데 TLS 라는 이름보다는 SSL 이라는 이름을 주로 사용합니다.

# 자세한 통신 방식
통신 과정을 살펴보면 아래와 같습니다.
1. 악수
2. 실제 데이터 전송
3. 세션 종료

#### 악수
1. 클라이언트가 서버에 접속하면서 **`클라이언트 헬로`** 를 한다. 클라이언트 헬로란 아래 과정이다.

    - 클라이언트 측에서 생성한 **`랜덤 데이터`** 를 서버로 전송
    - 클라이언트가 지원하는 **`암호화 방식 리스트`** 가 무엇인가를 서버쪽에 전송

2. 서버는 클라이언트에게 **`서버 헬로`** 를 한다. 서버 헬로란 아래의 과정이다.

    - 서버 측에서 생성한 랜덤 데이터를 클라이언트로 전송
    - 서버가 클라이언트의 암호화 방식 리스트 중 가장 안전한 방법을 선택 후 전송
    - 서버는 클라이언트에게 인증서를 전송

3. 클라이언트는 이 인증서가 브라우저에서 인정하는 인증 기관이 발급한 인증서인지를 확인한다. 만약에 브라우저에서 인정한 인증 기관이라면 **`공개키`** 는 이미 알고 있다.
4. 브라우저가 미리 알고 있는 **`공개키`** 로 복호화 한다.
5. 복호화 한 인증서 안에는 서버가 생성한 **`공개키`** 가 존재한다. 여기에서 서버의 **`공개키`** 를 획득한다.
6. 클라이언트와 서버가 랜덤키를 주고 받았는데 이를 조합해서 클라이언트는 **`프리 마스터 시크릿 키`** 를 생성한다. 그리고 이전 단계에서 얻은 인증서 안의 서버의 **`공개키`** 를 활용해서 **`프리 마스터 시크릿`** 키를 암호화 하고 서버에게 전송한다.
7. 서버는 **`비밀키`** 를 활용해서 복호화 한다. 여기까지 오면 클라이언트와 서버는 둘 다 **`프리마스터 시크릿 키`** 를 알고 있다. 이 값을 일련의 과정을 거쳐서 **`마스터 시크릿 키`**  로 만들고 **`세션 키`** 라는 것을 생성한다.
8. **`세션 키`** 를 대칭 키로 사용해서 데이터를 주고 받는다.
