# 객체지향 설계 과외 Day 2

리팩토링이란 동일한 기능에 설계를 변경하는 것 이다.

리팩토링만을 설명하는 책도 존재한다.

extract class -> 클래스를 분리하는 것

decompose conditional -> if 문 분기를 private method 로 빼는 것

캡슐화란 데이터는 보여주지 않고 메소드만 제공해주는 것

그런데 상속이 캡슐화를 깬다. 

그래서 상속은 해당 클래스가 상속을 받음으로써 해당 클래스가 되고 싶을 때 써야 한다.

해당 클래스의 기능이 필요할 때 쓰는게 아니다.

**아키텍처에 정답은 없지만 타당한 근거는 존재해야한다.**

다시 한번 설계 과정을 살펴보면
1. 문제 정의
2. 객체를 찾는다(시퀀스 다이어그램)
3. 퍼블릭 메소드를 연결한다(시퀀스 다이어그램)
4. 인터페이스를 만들면서 클래스 다이어 그램을 그린다.

팩토리 패턴으로 커플링을 루즈하게 만들 수 있다.

OCP 에서 비지니스 모델과 테크니컬 모델은 팩토리 패턴을 적용하는 것이 좋다.

플라이 웨이트는 메모리를 아끼기 위해서 객체를 여러개로 나누고 공통으로 사용되는 기능은 싱글톤 객체로 만드는 것.

예를 들어서 ㄱ 이라는 글자를 만드는데 폰트별로 전부 new 를 통해서 만들면 메모리가 감당할 수가 없음. 

ㄱ형태의 싱글톤 객체를 하나 띄우고 스타일을 바꿔야함.

템플릿 메소드 패턴은 실행과정이 똑같은 두개의 객체의 추상클래스를 만들어서 중복된 코드를 제거해준다.

브릿지 패턴은 ocp 의 ㄱ자 모델에서 비지니스 모델에서도 추상화를 통해서 갈아 끼울 수 있게 만드는 것.
