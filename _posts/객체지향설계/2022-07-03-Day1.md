UML 이란 Unified Model Language 이다.

설계의 과정(설계에는 정답은 없다)

1. FP 를 분석해서 어떤 객체들이 필요한지 알아낸다. 여기서 객체란 인터페이스가 아니다. 메모리를 할당해서 동작하는 **구체적인 객체**를 의미한다.

2. 그 다음 시퀀스 다이어그램을 통해서 객체들끼리 어떤 데이터를 주고 받는지 그려 본다.

3. **구체적인 구현체**들을 보면서 인터페이스를 만들면서 클래스 다이어그램을 설계한다.

인터페이스를 중심으로 만드는데 왜 이름이 클래스 다이어그램일까? 예전에는 인터페이스라는게 없었다고 한다. 

여기에서 재사용 될 수 있는게 무엇일까? 바로 시퀀스 다이어그램이 유일하다. 인터페이스는 재사용이 불가능하다. **구체적인 객체**만이 재사용이 가능하다.

시퀀스 다이어그램이나 클래스 다이어그램은 개발자들끼리 소통할 때 필요하다.

영업을 뛰는 사람에게는 패키지 다이어그램과 디플로이 다이어그램이 필요하다. 이들은 패키지 다이어그램을 통해서 실제 가치를 메길수도 있다.

컴포넌트 다이어그램 같은 경우 조금 더 큰 단위로 위로 보고할 때 유용하게 사용된다.

소프트웨어는 하나의 철학이다. extends 와 implentes 가 똑같이 override 되어서 사용하는 것 아니야? 라고 생각하면 아예 소프트웨어를 모르는 것 이다. 
extends 는 베놈을 생각하면 쉽다. 베놈에게 잡아먹히면 나는 인간이 아니라 베놈이 된다. 즉 책임이 바뀐다. -> **단일 책임의 원칙을 위배한다. 그리고 캡슐화가 깨지게 된다.**

인터페이스는 그렇지  않다. 단지 기능이 추가될 뿐이다.

따라서 상속을 쓸 일은 거의 없다. 아마 앞으로의 개발은 전부 interface 를 implements 한다고 생각하면 될 것 같다.

객체란 무엇일까?
클래스가 객체일까? 아니다. 클래스란 객체를 나타내는 여러가지들 중 하나일 뿐이다. 객체란 상태, 행위, 그리고 식별자까지 가지 원소로 이루어진 것 이다.
이 3가지를 표현할 수 있으면 객체라고 하는데 클래스, 프로토 타입 등등 이 존재한다.

<img width="523" alt="image" src="https://user-images.githubusercontent.com/52944973/177029250-73d9387c-d9e2-450d-a5ee-d1090c2dcac7.png">

A 라는 클래스가 보험 클래스이고 B 라는 객체가 csv 파일 객체라고 해보자.
A 클래스는 내부에서 `B b = new B();` 를 통해서 A가 B 를 의존하는 관계를 만들면 어떻게 될까?
A가 B 를 의존하고 있다는것은 B의 구현이 바뀌면 A에 문제가 생긴다는 것을 의미한다. 반면에 A의 코드가 바뀌어도 B 는 아무런 문제가 안된다. 의존하고 있지 않기 때문이다.
csv 파일은 콤마로 구분되어있는 문서인데 B 라는 문서를 읽기 위해 콤마를 전 처리하는 작업을 구현해야한다. 
만약 **csv 가 아니라 txt 를** 전달 받았다면 어떻게 될까? 이럴 경우 if else 구문으로 또 새롭게 구현해야한다. 가까운 미래에 A 클래스는 장황한 코드가 될 것이다.

이를 위해서 **의존성 역전**이 존재한다. 아래 그림과 같다.

<img width="559" alt="image" src="https://user-images.githubusercontent.com/52944973/177029348-378c3106-0fa1-4e07-a9d3-2ef2866a44e5.png">

A는 C라는 interface 를 뚫어둔다. 그 다음 아래와 같이 할당한다.  `C c= new B();` 그 다음 A 에서는 c.read() 를 구현하는 코드를 작성하면 된다. c.read() 메소드가 어떻게 동작 하는지는 A의 관심 밖이다. C 의 구현체들이 각자 알아서 신경쓰면 된다. 
B 의 값이 바뀌었다고 해서 A 에 문제가 생기는가? 전혀 아니다. A 가 B 에 의존하던 상황에서 B 가 C 에 의존하는 상황이 되었다. 
이를 **의존성 역전** 이라고 한다.
SOLID 원칙에서 D 에 해당한다.

OCP 는 인터페이스로 정의한 기능들을 B가 구현해야한다는 것이다. B 는 인터페이스의 기능을 변경할 수 는 없다 .하지만 어떻게 구현 하는지는 B 의 마음이다. 확장에는 열려있다. 인터페이스를 구현함으로써 이를 강제할 수 있다.

LSP 는 인터페이스에서 정의한 이외의 Public 메소드를 B가 가지고 있으면 안된다는 것이다. 오리를 예로 들어보자. 천둥오리와 장난감 오리가 존재한다. 둘다 오리를 구현해서 만들었다. 인터페이스 사용자가 오리의 울기 라는 메소드를 사용했다고 해보자. 천둥오리는 잘 운다. 그런데 장난감 오리가 울지 않는다. 이유가 무엇일까? 장난감 오리는 public 메소드로 선언된 전기를 공급받는다 라는 메소드가 선행되어야하기 때문이다. 인터페이스 사용자는 아무리 울기를 실행해도 이를 해결할 수 없다. 이러한 이유로 LSP 원칙이 중요하다.

TDD 가 중요한 이유도 이러한 이유이다. 보통 외부 서비스 같은 경우 mock 객체로 만들어서 사용한다. 다양한 이유가 있겠지만 병렬적으로 개발하기 위해서 그렇다고 가정해보자. 외부 서비스 개발이 완성 되어서 내 서비스에 붙이려고하는데 동작을 안한다. 아마 LSP 를 어겼기 때문일 확률이 높다. TDD 는 LSP 의 문제를 어느정도 잡아줄 수 있기 때문에 중요하다.
